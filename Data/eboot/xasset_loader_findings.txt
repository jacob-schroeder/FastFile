Sound loader block map from common_mp investigation
- Core stream helpers: 0x11da90 pushes the block id in r3, 0x11db00 pops, 0x11db50 aligns the active block by mask r3, and 0x11dd00 is Load_Stream(size in r5, target pointer in r4). Load_Stream with r3=1 consumes/copies source bytes and advances the active block; r3=0 resolves an existing cell without consuming source bytes.
- Offset patch helpers differ by pointer kind: 0x11dc00 patches a cell to base+offset and is used by XString/direct-address pointers; 0x11dbd0 patches a cell by dereferencing a prior pointer cell and is used by wrapper loaders such as snd_alias_list_t*, SndCurve*, and LoadedSound*.
- Block ids observed in these loaders: TEMP=0, PHYSICAL=1, LARGE=4, VERTEX=6.
- 0x116328 dispatch type 0x0a (Sound) to 0x10a9d8, not directly to 0x10a900. 0x10a9d8 is the snd_alias_list_t* wrapper: Load_Stream 4 bytes for the asset pointer cell, push TEMP (0), align via 0xf2e18, then call 0x10a900 for inline/insert roots. Non-inline offsets use 0x11dbd0.
- 0x1167c0 (`Load_XAssetList`) pushes LARGE (4) around the top-level script-string and XAsset table families. The nested script-string root temporarily pushes TEMP (0), but the `XAsset` row array itself is consumed by 0x116738 while LARGE is still active, so the serialized asset table payload block is LARGE rather than TEMP.
- 0x1108f8 top-level `scriptStrings` root uses 0xf3d50 (`DB_AllocStreamPos(3)`) before 0x110870, so the pointer table itself is LARGE-aligned to 4 bytes. The per-entry inline strings then use 0x110760 -> 0xf3d80 (`DB_AllocStreamPos(0)`), so each string begins at the current LARGE cursor with no per-string alignment; only the next top-level family realigns.
- 0x1167c0 top-level `assets` root uses 0xedfb8 (`DB_AllocStreamPos(3)`) before 0x116738, so the first `XAsset` row starts at the next 4-byte LARGE boundary after the script-string bytes. In `common_mp`, the last script string (`\"psphere1\\0\"`) ends at LARGE:0x5EAD and the asset table begins at LARGE:0x5EB0 after three pad bytes.
- 0x10a900 is the snd_alias_list_t body: Load_Stream 0x0c, push LARGE (4), Load_XString at +0x00, then resolve +0x04 aliases. Inline aliases align via 0xf2e48 and 0x10a878 consumes count(+0x08) * 0x64 entries.
- 0x10a600 is the snd_alias_t body: Load_Stream 0x64, Load_XString at +0x00,+0x04,+0x08,+0x0c,+0x10 in the current LARGE context. It pushes TEMP (0) only around +0x14 sound-file array resolution, then pops before resolving +0x50 and +0x60.
- 0x10a578 consumes the +0x14 sound-file array: count * 0x10 SoundFile entries in TEMP, each through 0x10a4a8. The count helper at 0x116f08 normally returns 1.
- 0x10a4a8 is the SoundFile body: Load_Stream 0x10 in TEMP, then push LARGE (4) for the +0x04 union. 0x10a468 dispatches loaded entries through 0x10a030 and streamed/primed entries through 0x10a420/0x10a3d8 style wrappers.
- 0x10a030 is the LoadedSound* wrapper: Load_Stream 4 bytes for the union pointer cell, push TEMP (0), align via 0xf2ff8, then call 0x109fc0 for inline/insert roots. Non-inline offsets use 0x11dbd0.
- 0x109fc0 is the LoadedSound body: Load_Stream 0x1c in TEMP, push LARGE (4), Load_XString at +0x00, then call 0xf62d0 for the embedded MssSound fields.
- 0xf62d0 runs while LARGE is active for the sound-info fields. It aligns +0x10 seek-table payload via 0xf5408 (4-byte alignment) and consumes lhz(+0x0e) * 4 bytes through 0xec970. It then pushes PHYSICAL (1), aligns +0x14 audio payload via 0xf54f8 (64-byte alignment), consumes the physical byte count through 0xecec8, and pops PHYSICAL.
